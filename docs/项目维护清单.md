# 项目维护清单

## 一、当前重点文件

| 文件/目录 | 用途 |
| --- | --- |
| `README.md` | 项目总览、常用命令、发布流程 |
| `docs/项目链接整理.md` | 审核提交和线上链接统一入口 |
| `public/support.html` | 对外支持页和安装包下载按钮 |
| `public/privacy.html` | 对外隐私政策 |
| `public/installer-exit-codes.html` | 安装失败返回码说明 |
| `public/downloads/v1.0.0/` | 静态托管安装包目录 |
| `cloudbaserc.json` | CloudBase 部署环境和框架配置 |
| `src/components/PrivacyPolicy.tsx` | 应用内隐私政策组件 |

## 二、发布前检查

1. 执行 `npm run build`，确认 TypeScript 和 Vite 构建通过。
2. 确认 `dist/downloads/v1.0.0/jin-bang-zhi-yi-setup-1.0.0.exe` 存在。
3. 执行 `npx tcb hosting deploy dist -e sjy-d0gxtaklr8e1be761` 发布静态资源。
4. 验证以下链接均返回 `200`：
   - `https://jinbangzhiyi.online/`
   - `https://jinbangzhiyi.online/privacy.html`
   - `https://jinbangzhiyi.online/support.html`
   - `https://jinbangzhiyi.online/installer-exit-codes.html`
   - `https://jinbangzhiyi.online/downloads/v1.0.0/jin-bang-zhi-yi-setup-1.0.0.exe`

## 三、容易混淆的点

- 正式审核链接统一填写 `https://jinbangzhiyi.online/` 及其对应页面路径。
- CloudBase API 端点不是软件包下载地址，不要填到“软件包 URL”字段。
- `release/` 是 Electron 打包输出目录，`public/downloads/` 是 Web 静态托管下载目录，两者用途不同。
- 安装包超过 100 MB，不适合直接推送到 GitHub 普通仓库。
- `dist/` 是构建产物，部署 CloudBase 用它，但不要手工改里面的文件；源文件应改 `src/` 或 `public/`。

## 四、建议后续整理

1. 将安装包长期放到对象存储或 Release 系统，避免仓库内维护大文件。
2. 如果后续更换证书或域名，先验证 DNS、证书和下载直链，再批量替换文档和页面链接。
3. 梳理 `.env` 中的密钥，确认只保留前端可暴露的 publishable key，服务端密钥不要进入前端构建。
4. 统一检查中文编码，避免 PowerShell 控制台显示乱码影响维护判断。
